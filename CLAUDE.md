## Environment setup

Copy `.env.example` to `.env` and fill in:
- `VITE_SUPABASE_URL` and `VITE_SUPABASE_KEY` — from Supabase Dashboard → Settings → API
- `VITE_MP_PUBLIC_KEY` — MercadoPago public key (client-side); prefix `TEST-` means sandbox mode
- `MP_ACCESS_TOKEN` — MercadoPago access token (Vercel env, no `VITE_` prefix)
- `RESEND_API_KEY` — Resend email API key (Vercel env); omit to skip emails silently
- `RESEND_FROM_EMAIL` — sender address (defaults to `Level Up <onboarding@resend.dev>`)
- `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` — used by Vercel Edge Functions (webhook) to bypass RLS
- `MP_WEBHOOK_SECRET` — optional; when set, the webhook verifies MP's HMAC signature

## Architecture

**Stack:** React 18 + Vite + TypeScript + Tailwind CSS + shadcn/ui + Supabase + MercadoPago + Resend

This is a Chilean e-commerce SPA for supplements, wearables, and wellness products. Deployed on Vercel.

### Provider tree

`QueryClientProvider` (TanStack Query) → `TooltipProvider` → `BrowserRouter` → `Navbar + Routes + Footer`

`AuthProvider` is **not** app-wide — it wraps only the `ProductosAdmin` route in `src/App.tsx`. The rest of the app is unauthenticated.

### Path aliases

`@/` maps to `src/`. All internal imports use this alias.

### Data flow

- **Products** live in Supabase table `products`. `src/lib/dbProducts.ts` is the single DB access layer; `src/lib/products.ts` re-exports `fetchAllProducts` / `fetchProduct` as the public API (and also re-exports `formatCLP`). Pages use `useProducts()` / `useProduct(id)` hooks from `src/hooks/useProducts.ts`, which cache for 60 seconds via TanStack Query. Product `id` is a **text slug** (not a UUID), generated by `slugify(name)` in the admin UI.
- **Cart** is persisted in `localStorage` under key `cjhealth_cart`. All mutations dispatch a `cart-updated` window event so the Navbar badge updates reactively. Cart logic lives entirely in `src/lib/cart.ts`; the `useCart()` hook in `src/hooks/useCart.ts` subscribes to that event.
- **Auth** is handled by Supabase Auth (email/password + Google OAuth). `AuthContext` in `src/contexts/AuthContext.tsx` provides `{ user, session, loading, signIn, signUp, signInWithGoogle, signOut }`. Access it via `useAuth()` hook from `src/hooks/useAuth.ts`.
- **Orders** are stored in Supabase tables `orders` and `order_items`. RLS policies allow anonymous inserts (guest checkout) and restrict reads to the owning user or anon (filtered client-side by orderId).
- **Stock validation** happens at checkout via `src/lib/checkout.ts → validateCartStock()`, which queries current stock for all cart items in one request and returns per-item errors.

### Order status lifecycle

| Status | Set by | Meaning |
|---|---|---|
| `pendiente_pago` | Client (MP flow) | Order created, awaiting payment |
| `pendiente_verificacion` | Client (transferencia flow) | Proof uploaded, awaiting manual review |
| `verificado` | MP webhook (authoritative) or client at `/pago/exito` | Payment confirmed |
| `cancelado` | Client at `/pago/error` | Payment failed |

For the MP flow, `/pago/exito` also does a client-side `status → verificado` update optimistically; the MP webhook is the authoritative source and runs independently.

### Payment flow

The checkout journey: `/checkout` (customer data) → `/pago/mercadopago` (always the first step) → optionally `/pago/transferencia`. Two paths from `/pago/mercadopago`:

1. **MercadoPago** — calls `src/lib/mercadopago.ts → createPreference()`, which POSTs to the Vercel Edge Function `api/mercadopago/create-preference.ts`. The order is created in Supabase **before** redirecting to MP. `sessionStorage.pending_order_id` bridges the external MP redirect back to `/pago/exito`. Use `isMPTestMode()` to check if `VITE_MP_PUBLIC_KEY` is a test key and redirect to `sandbox_init_point` instead. Stock is decremented by the webhook on `approved` payment.
2. **Transferencia bancaria** (`/pago/transferencia`) — user uploads a payment proof file stored in Supabase Storage bucket `comprobantes`. Stock is decremented **client-side** via `supabase.rpc("decrement_order_stocks")` at order creation (no webhook involved).

### Vercel Edge Functions (`api/`)

All files under `api/` run as Vercel Edge Functions (`export const config = { runtime: "edge" }`):
- `api/mercadopago/create-preference.ts` — creates MP preference; strips `payer` from client payload to prevent card-leak across users; injects `back_urls` server-side.
- `api/mercadopago/webhook.ts` — receives MP payment notifications; verifies HMAC signature when `MP_WEBHOOK_SECRET` is set; on `approved` payment: updates order status to `"verificado"`, calls `decrement_order_stocks` RPC (idempotent), then sends order emails.
- `api/emails/send-order-confirmation.ts` — manually trigger order confirmation emails.
- `api/_lib/emails.ts` — shared Resend email logic; sends customer confirmation + admin notification via `Promise.allSettled` (one failure never blocks the other).

### Email notifications

`sendOrderEmails()` in `api/_lib/emails.ts` fires on two occasions: (1) when the MP webhook confirms an approved payment, and (2) for transferencia orders via a fire-and-forget `fetch` to `api/emails/send-order-confirmation.ts` from the client. Silently skips if `RESEND_API_KEY` is not set.

### Admin area

`/admin/productos` → `src/pages/admin/ProductosAdmin.tsx`. No client-side auth guard — access control enforced entirely by Supabase RLS: only `cjhealthsupply@gmail.com` can write to `products` or upload to `product-images`. The component renders a "forbidden" state when the logged-in user is not that email.

### Database schema

Run `supabase_setup.sql` in Supabase SQL Editor to create all tables, RLS policies, and the `decrement_order_stocks` function. Run `supabase_seed_products.sql` to populate initial product data.

- `products` — catalog (slug `id`, `category`, `section`, `stock` integer, `active` for soft-delete, `details` as JSONB array)
- `orders` — customer orders (`user_id` nullable for guest checkout, `status` defaults to `'pendiente_pago'`, `stocks_decremented` boolean for idempotent stock decrement, `payment_proof_url` for transferencia)
- `order_items` — line items per order

Product images → `product-images` Storage bucket (public). Payment proofs → `comprobantes` (public).

The `decrement_order_stocks(p_order_id uuid)` RPC is `SECURITY DEFINER` and idempotent (checks `stocks_decremented` flag before acting).

### Routing

All routes are defined in `src/App.tsx`. The `vercel.json` rewrite rule sends all non-`/api/` paths to `index.html` for SPA routing.

### Currency

All prices are stored as integers in CLP (Chilean pesos). Use `formatCLP()` from `src/lib/cart.ts` (also re-exported from `src/lib/products.ts`) for display formatting.
